# Do's and Don'ts

This file provides practical guidance for creating Lumina visuals.

It should be used as a quick design review checklist for illustrations, presentation graphics, web imagery, diagrams and AI-generated image prompts.

The goal is not to restrict creativity.

The goal is to protect the Lumina visual identity.

---

## Core Rule

A Lumina visual should make knowledge, people and intelligent library services easier to understand.

If a visual looks impressive but does not improve understanding, simplify it.

If a visual looks technically advanced but feels generic, make it more human-centred and knowledge-centred.

If a visual looks exciting but not trustworthy, reduce the visual intensity.

---

# Do

## Show Knowledge as the Main Subject

Use visual elements that represent academic knowledge and library services.

Good elements include:

- documents
- metadata
- collections
- research outputs
- search results
- subject labels
- knowledge graphs
- semantic clusters
- data cards
- workflows
- reusable services

Knowledge should be visible and structured.

Technology should help organise, connect or enrich that knowledge.

---

## Show People with Expertise

People may appear in Lumina visuals when they clarify purpose, responsibility or collaboration.

Show people as:

- librarians
- researchers
- students
- developers
- data specialists
- service owners
- domain experts

They should appear calm, focused and professional.

They should interact with knowledge, interfaces or workflows in a meaningful way.

People should represent expertise, judgement and interpretation.

---

## Use Abstract Interfaces

Use simplified interface elements to explain concepts.

Good interface elements include:

- search bars
- result cards
- metadata panels
- dashboards
- workflow steps
- status cards
- filter panels
- API cards
- review checkpoints
- small charts
- semantic graph panels

Interfaces should look plausible, but not like real product screenshots.

They should explain the idea rather than reproduce a finished software product.

---

## Visualise Relationships

Lumina is about connected knowledge.

Use relationships to explain meaning.

Good relationship elements include:

- clean connection lines
- semantic clusters
- grouped document cards
- linked metadata tags
- directional flows
- knowledge graph nodes
- reusable service layers
- highlighted pathways

Connections should be readable and purposeful.

Every line should have a reason.

---

## Use ETH Colours with Restraint

Use the Lumina colour system consistently.

Prefer:

- white backgrounds
- black text-like elements
- ETH Blue as the primary colour
- ETH Petrol for technical and semantic relationships
- ETH Grey for supporting elements
- accent colours only when they add meaning

A Lumina visual should feel calm before it feels colourful.

---

## Keep Compositions Calm

Use generous whitespace.

Create one clear focal point.

Limit the number of visual elements.

Use hierarchy to guide the viewer.

Allow the viewer to understand the image quickly.

Good Lumina visuals should feel structured, spacious and intentional.

---

## Make Complexity Understandable

Lumina deals with complex topics such as AI agents, embeddings, semantic search, metadata enrichment, APIs and knowledge graphs.

Do not show complexity as chaos.

Instead, simplify it through:

- layers
- cards
- clusters
- flows
- modules
- panels
- review steps
- clear groupings

The visual should help people understand the concept.

---

## Use AI as an Enabling Layer

AI should usually be represented indirectly.

Good AI metaphors include:

- semantic recommendations
- enriched metadata
- highlighted relationships
- suggested connections
- workflow assistance
- intelligent review support
- pattern recognition
- contextual discovery

AI should feel useful, subtle and responsible.

---

## Leave Room for Text

Many Lumina visuals will be used in presentations and web layouts.

Leave space for:

- slide titles
- headings
- captions
- callouts
- explanatory text
- interface labels added later

Do not fill the entire canvas.

Important text should usually be added later in PowerPoint, Figma or the web layout.

---

# Don't

## Don't Use Robots as the Default Symbol for AI

Avoid humanoid robots, robot assistants or machine characters.

Lumina is not about replacing people.

Lumina is about supporting people with intelligent services.

Robots make the visual language feel generic and misleading.

---

## Don't Use Glowing Brains

Avoid glowing brains, artificial brains, neural heads or similar AI clichés.

They feel generic and often overpromise what the system does.

Lumina should represent AI through useful knowledge work, not through science-fiction symbolism.

---

## Don't Use Cyberpunk or Sci-Fi Aesthetics

Avoid:

- neon lights
- dark futuristic control rooms
- holograms
- glowing grids
- dramatic blue light
- surveillance-style dashboards
- sci-fi interfaces
- digital tunnels
- matrix-like backgrounds

Lumina should feel academic, calm and trustworthy.

It should not feel like science fiction.

---

## Don't Use Binary Code Backgrounds

Avoid backgrounds filled with code, numbers or binary patterns.

They make visuals look generic and technical without adding meaning.

If technical structures are needed, use abstract flows, cards, APIs or semantic links instead.

---

## Don't Make Dashboards Too Dense

Avoid overloaded dashboards with many tiny panels, unreadable tables or excessive charts.

Dashboards should communicate overview and control.

They should not create visual stress.

Use only a few clear panels and status elements.

---

## Don't Use Rainbow Palettes

Avoid colourful gradients or many unrelated colours.

Do not use colour for decoration only.

Lumina should not look like generic SaaS marketing.

Use colour to create meaning, hierarchy and identity.

---

## Don't Create Fake Product Screenshots

Avoid visuals that look like exact screenshots of real software unless explicitly requested.

Generated screenshots often contain unreadable or misleading UI text.

Use abstract interfaces instead.

Real product screens should be captured and designed separately.

---

## Don't Rely on Generated Text

Image generation models often produce incorrect or unreadable text.

Avoid asking the image model to create important labels, titles, interface text or logos.

Use abstract text lines or short placeholder labels only.

Add important text later in the final design tool.

---

## Don't Overload the Image with Icons

Avoid using many small icons to represent every concept.

Too many icons make visuals feel decorative and shallow.

Prefer fewer, more meaningful visual structures.

Use icons only when they clarify the concept.

---

## Don't Make People Decorative

Do not add people just to make an image feel friendly.

People should have a role in the visual story.

They should review, explore, connect, organise, monitor, interpret or collaborate.

If the person does not add meaning, remove them.

---

## Don't Use Generic Corporate Stock Scenes

Avoid:

- people shaking hands
- people pointing at random charts
- business suits in glass offices
- generic team meetings
- exaggerated success poses
- smiling characters without context

Lumina visuals should feel specific to academic knowledge work and library services.

---

## Don't Show Infrastructure as Server Rooms

Avoid server racks, cables, cloud clutter and data centre imagery unless the topic explicitly requires infrastructure.

Lumina should be visualised through knowledge services, APIs, workflows and semantic systems.

The value is not the hardware.

The value is the intelligent service layer.

---

## Don't Make AI Look Magical

Avoid magic sparkles, glowing transformations or miraculous automation.

Lumina should communicate responsible AI-supported workflows.

If a process is automated, also show structure, review, quality or human oversight where appropriate.

---

# Topic-Specific Guidance

## Semantic Search

Do:

- show search interfaces
- show result clusters
- show semantic relationships
- show connected document cards
- show discovery beyond keywords

Don't:

- rely only on a magnifying glass
- show endless result lists
- show chaotic networks

---

## Metadata Enrichment

Do:

- show documents becoming more structured
- show tags, labels, annotations and layers
- show quality review
- show before-and-after structures

Don't:

- show metadata as a boring spreadsheet only
- show enrichment as magic
- imply that human expertise is irrelevant

---

## Knowledge Graphs

Do:

- show readable node clusters
- connect documents, topics, people or collections
- keep the graph selective and meaningful

Don't:

- create dense network spaghetti
- show random connections
- use graphs as decoration only

---

## AI Agents

Do:

- show tasks, workflows and assistance
- show agents as abstract service modules
- show orchestration and review
- show collaboration between people and systems

Don't:

- show agents as robots
- show autonomous machine characters
- imply uncontrolled automation

---

## APIs and Data Flows

Do:

- show reusable service blocks
- show clean connections
- show data moving between systems
- show APIs as cards, ports or service interfaces

Don't:

- show chaotic technical diagrams
- overuse cloud icons
- show infrastructure complexity without purpose

---

# Quick Review Checklist

Before using or publishing a Lumina visual, check:

- Does it have one clear main idea?
- Does it centre knowledge, people or library services?
- Does technology support the story?
- Does it use ETH Blue, black and white as the main colour basis?
- Are secondary and accent colours used with restraint?
- Is the composition calm and readable?
- Are relationships meaningful and easy to follow?
- Are AI clichés avoided?
- Is the visual suitable for presentations or web use?
- Does it feel like part of the Lumina ecosystem?

If several answers are no, simplify and revise the visual.
