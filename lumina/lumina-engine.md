# Lumina Engine

Lumina Engine is the processing and intelligence layer of the Lumina ecosystem.

It transforms library data, documents, metadata and semantic signals into enriched, connected and reusable knowledge structures.

Lumina Engine should be visualised as active, intelligent and structured.

It is not a user interface.

It is the part of Lumina where data is processed, enriched, matched, classified and prepared for intelligent library services.

---

## Purpose

Lumina Engine supports the transformation of raw or fragmented information into structured, enriched and semantically useful knowledge objects.

It may support:

- metadata enrichment
- embedding generation
- embedding-based matching
- automated subject indexing
- semantic search preparation
- classification support
- entity extraction
- relationship detection
- recommendation signals
- quality checks
- data transformation
- AI-assisted processing workflows
- preparation of data for discovery and reuse

Lumina Engine should feel like a responsible intelligence layer.

It helps library data become more meaningful, connected and useful.

---

## Core Message

Lumina Engine turns data into enriched knowledge.

The key message is:

**Lumina Engine processes, enriches and connects library information so that intelligent services can be built on top of it.**

It should communicate:

- transformation
- enrichment
- semantic understanding
- automation with structure
- data quality
- intelligent support
- reusable outputs
- responsible processing

Lumina Engine should not feel like uncontrolled automation.

It should feel like a structured and trustworthy processing environment.

---

## Role in the Lumina Ecosystem

Lumina Engine works closely with Lumina Core.

Lumina Core provides the trusted foundation, shared data and service access.

Lumina Engine performs processing, enrichment and semantic computation.

The outputs of Lumina Engine may be used by:

- Lumina Core
- Lumina Studio
- Lumina Experience
- search systems
- metadata services
- subject indexing workflows
- API-based services
- discovery interfaces
- quality review processes

Lumina Engine should be visualised as the active layer that creates semantic value from library data.

## Visual Story

The visual story of Lumina Engine should be about transformation from information to meaning.

Possible story:

Document cards, metadata records and research outputs enter a structured processing flow. Inside the Engine, embeddings, semantic matching, classification, enrichment and quality checks are applied. The output is a set of cleaner, richer and more connected knowledge objects that can be reused by Lumina services.

The visual should show:

- input data entering a processing flow
- enrichment steps or service modules
- semantic relationships being discovered
- metadata becoming more structured
- subject labels or classifications being suggested
- embeddings or semantic clusters being formed
- outputs becoming reusable services or enriched knowledge objects

The viewer should understand that Lumina Engine does not simply store data.

It actively improves, structures and connects it.

## Preferred Composition

Lumina Engine works especially well in process-oriented compositions.

Preferred layouts:

### Enrichment Pipeline

Input objects enter from the left, pass through several processing modules, and become enriched outputs on the right.

Good for metadata enrichment, subject indexing and classification workflows.

### Semantic Matching Scene

Documents, subject labels or knowledge objects appear as clusters in a semantic space. Matching connections show which objects are related.

Good for embedding-based matching and automated subject indexing.

### Processing Layer

Lumina Engine appears as a middle layer between raw data and user-facing services.

Good for system overviews and architecture explanations.

### Before-and-After Structure

Raw or sparse metadata appears on one side. Enriched, connected and structured metadata appears on the other.

Good for communicating value to non-technical audiences.

### Modular Service View

Engine functions appear as modular service blocks such as embeddings, classification, enrichment, matching and validation.

Good for technical or architectural presentations.

## Colour Emphasis

Use:

- ETH Blue for the main Engine layer or central processing elements
- White for background and calm spacing
- Black for structural contrast and text-like elements
- ETH Petrol for data flows, semantic links and embedding relationships
- ETH Grey for secondary modules and neutral processing steps

Accent colours may be used sparingly:

- ETH Green for successful enrichment, validation or completed processing
- ETH Purple for advanced AI capabilities, embeddings or semantic intelligence
- ETH Bronze for subject categories or classification domains

Avoid ETH Red unless showing errors, warnings or failed quality checks.

Lumina Engine visuals may be slightly more dynamic than Lumina Core visuals, but they should still remain calm and structured.

## Relationship to Other Components

### Lumina Core

Lumina Core provides the trusted data and service foundation.

Lumina Engine processes and enriches data from Core or prepares outputs for Core.

Visual relationship:

Core may appear as the foundation or shared service layer, while Engine appears as the active processing layer connected through clean data flows and APIs.

---

Lumina Engine performs the processing that Studio may observe, configure or validate.

Visual relationship:

---

Lumina Experience uses enriched data and semantic structures to support discovery and interaction.

Lumina Engine creates or prepares many of the semantic signals that Experience can use.

Visual relationship:

Experience may appear as a user-facing interface receiving enriched knowledge objects, recommendations or semantic search capabilities from Engine.

## Specific Topic: Embedding-Based Matching for Subject Indexing

For embedding-based matching, show a semantic space rather than a technical vector database.

Suitable visual elements:

- document cards
- subject heading cards
- clusters of related concepts
- soft connection lines
- similarity paths
- highlighted best matches
- suggested subject labels
- human review checkpoint
- enriched metadata output

The visual should communicate that the system identifies meaningful relationships between documents and subject concepts.

Avoid showing mathematical formulas, vector coordinates or neural-network diagrams unless the audience is explicitly technical.

For management or library audiences, focus on semantic proximity, quality and human review.

For technical audiences, a more schematic embedding layer may be included, but it should remain visually calm.

---

## Avoid

Do not show Lumina Engine as:

- a literal combustion engine
- gears or machinery
- a factory automation line
- a black-box AI oracle
- a robot doing library work
- a glowing brain
- a chaotic neural network
- a server room
- a cyberpunk processing centre
- a magical transformation effect
- an autonomous system without human accountability

Avoid implying that AI enrichment is perfect, mysterious or fully automatic.

Where appropriate, show quality checks, review points or structured outputs.
